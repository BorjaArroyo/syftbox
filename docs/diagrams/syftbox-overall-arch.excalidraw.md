SyftBox syncs files globally through a shared folder structure (~/SyftBox) where each user/organization owns and administers their top-level directory (~/SyftBox/datasites/<user_identifier>/), called a "datasite."

Core Architecture:
  - Decentralized control: No central authority; each datasite owner sets ADMIN/WRITE/CREATE/READ permissions for their directory
  - Client (Go program): Monitors local changes, syncs with server bidirectionally
  - Server (cloud/S3): Stores complete filesystem copy of the whole network in an S3 bucket, validates permissions, coordinates sync operations between clients ^1FQen1Qw

Despite its decentralized vision, the current implementation has critical centralization issues:

  Single Point of Failure:
  - Monopoly control: OpenMined runs the only server and could shut down the network anytime
  - Protocol dictatorship: OpenMined exclusively designs the server software
  - Financial dependency: Network survival depends solely on OpenMined's funding

  Security/Privacy Vulnerabilities:
  - No data privacy: Server operators can view all files/folders (can be solved with e2e encryption)
  - Permission bypass: Server operators can override any user-set permissions
  - Vendor lock-in: Clients can only connect to one server, creating total dependency

  These limitations fundamentally contradict the decentralized power model SyftBox aims to achieve. ^WECiKHPA
